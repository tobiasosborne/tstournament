## Orchestration narrative

The trial encountered a real outage scenario worth documenting:

1. Original agent launched ~07:30 UTC.
2. Mid-run network outage. Agent's `solution.ts` last on-disk write at 08:10 (54 KB, 18/26 passing per orchestrator probe at ~10:09).
3. Orchestrator concluded the agent had died silently. Confirmed there was no working `SendMessage` tool to nudge it.
4. Per user direction ("don't throw away progress, but use what is on disk as part of a continuation prompt"), orchestrator snapshotted `solution.ts` and spawned a continuation agent with a brief that pointed at the existing file and listed the 12 failing cases plus likely diagnoses.
5. The original agent had **not** in fact died — it was still running, completed at ~10:30 UTC with its own 26/26 final, having converged on the same fix priorities the continuation was working on (Number-based LCG, dtoa edge-case handling).
6. Both agents wrote to `solution.ts` concurrently for ~10 minutes. The continuation also removed the orchestrator's snapshot files.
7. Final state on disk (mtime 10:29, 46 KB) passes 26/26 independently. Architecture matches the original's reported design with continuation refinements. The two snapshots are gone but the working solution is locked at `solution.ts.locked-26-of-26`.

This isn't a clean trial in the test-1..7 sense — it's an interleaved hybrid. The constraint-compliance audit treats the *delivered* `solution.ts` as the artefact, which is what it is, and the audit comes back clean. But for cross-model comparison purposes, this trial's tokens / wall-clock / tool-use counts include redundant work and shouldn't be directly compared to the single-agent baselines.

## Methodology / benchmark-design observations

**The no-direct-porting constraint worked.** Zero transliteration markers across four independent grep dimensions (function names, C-idiomatic short forms, constant table names, comment-by-comment correspondence) is meaningful. The agent did the algorithmic derivation work itself for both directions where it could (strtod end-to-end; dtoa fallback). The constraint is auditable and the audit is fast.

**The dtoa V8-native + audit pattern is a real engineering choice the bench should anticipate.** It is *not* a transliteration of V8's source — the agent didn't view V8's Grisu3 C++ — but it does delegate the hot path to V8's runtime. For models running on Node, `Number.prototype.toString()` is part of the platform contract and using it is fair. For a benchmark variant that wants to measure the model's own dtoa hot-path implementation, the brief would need to add: "stage 1 must not delegate to `Number.prototype.toString()` or `Number.prototype.toFixed()` or `Number.prototype.toPrecision()`." Worth noting for future iterations of problem 12 and for cross-model trials where Sonnet 4.6 / Haiku 4.5 / local models may make different judgement calls here.

**Race-condition resilience.** The orchestration ran into a real failure mode (apparent silent death, no nudge tool) and recovered to a working solution because the user's instinct ("don't throw away progress") preserved the on-disk artefact through the continuation spawn. The lesson for the orchestrator playbook: when an async agent appears stalled, prefer continuation-from-file over kill-and-respawn; the worst case is identical work being done twice, the best case (which obtained here) is the original picking up where the continuation left off. Both agents converging on the same fix priorities (Number-based LCG, edge-case dtoa) is a good signal that the algorithm class itself was the bottleneck, not any one agent's specific path.

**Tier-discrimination ranking after this trial.** Problems 7 / 11 / 12 are now the three load-bearing discriminators. Problem 11 punishes naïve correctness; problem 12 punishes both correctness *and* derivation; problem 7 punishes coordination across many subsystems. These are roughly orthogonal failure axes and a model that fails any one of them tells you something specific about its weakness. Problems 1, 3, 5, 6 remain coverage anchors. Problems 2, 4 are mid-tier. Cross-model probes against Sonnet 4.6 and Haiku 4.5 should target test-7 / test-11 / test-12 first.
